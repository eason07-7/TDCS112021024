# 教師對照操作指南 — AWS Console 手動展示流程

> 本指南讓老師當場在 AWS Console 看到本專題的所有實證。**全程在 AWS 雲端、不碰本機檔**。
>
> **執行方式**：學生開螢幕共享、跟著本指南一步一步操作、念對應講稿給老師聽。
>
> **總耗時**：12-15 分鐘（共 10 步、每步 1-2 分鐘）。
>
> **準備**：學生先刷新 Learner Lab 憑證、登入 AWS Console、確認右上角 region 是 **us-east-1 (N. Virginia)**。

## 步驟 6：Athena 當場跑一次查詢（最強的「現場可驗證」證據）

### 6.1 操作

1. 上方搜尋列輸入 **`Athena`** → 點 Amazon Athena
2. 進「Query editor」
3. 上方 Workgroup 下拉選單、選 **`tdcs-dl-wg`**（不是預設的 primary）
4. 中間 Query 編輯區、貼入下面 SQL：

```sql
SELECT yyyymm,
       COUNT(*) AS rows,
       SUM(counts) AS total_traffic
FROM tdcs_dl.cleaned_v2_skeleton
GROUP BY yyyymm
ORDER BY yyyymm;
```

5. 點下方藍色按鈕 **「Run」**
6. 等大約 1 秒

### 6.2 看什麼

| Status | 應顯示 |
|---|---|
| Run time | 約 800-1500 ms |
| Data scanned | 約 30-60 KB |
| 結果 | 3 行：202603 / 202604 / 202605 |

結果應為：

| yyyymm | rows | total_traffic |
|---|---|---|
| 202603 | **14,116** | 1,529,833 |
| 202604 | **11,384** | 1,209,280 |
| 202605 | **7,066** | 760,353 |

### 6.3 講稿

> 「**14,116 這個數字、從書面報告封面到 §4.1 反覆出現很多次、就是麻豆段 3 月份的清洗結果**。**現在我們在 AWS Athena 上面、當場再跑一次同樣的 SQL、結果一模一樣**。
>
> 三個月加起來 32,566 行、總車流大概 350 萬次通過。**scan 才幾十 KB**——因為 Athena 用 Parquet 列式儲存、只讀需要的欄位、所以這麼快、這麼便宜。
>
> 這個查詢老師之後想再跑、用學生帳號隨時可以再跑、每次跑都會給同樣答案。」

## 步驟 9：SQS 看 broker 配置（visibility timeout 920 秒）

### 9.1 操作

1. 上方搜尋列輸入 **`SQS`** → 點 Amazon Simple Queue Service
2. 在 Queues 清單找 **`tdcs-dl-clean-jobs`**、點進去
3. 看「Details」分頁

### 9.2 看什麼

| 欄位 | 應顯示 |
|---|---|
| Type | Standard |
| Visibility timeout | **15 min 20 sec（920 秒）** |
| Dead-letter queue | 指向 `tdcs-dl-clean-jobs-dlq` |
| Maximum receives | 2 |

切到「Lambda triggers」分頁、應看到觸發 **`tdcs-dl-cleaner`** Lambda。

### 9.3 講稿

> 「SQS 就是 §3.2 那個解決 30 秒對 847 秒鴻溝的關鍵。
>
> **Visibility timeout 設 920 秒、比 Lambda 的 900 秒上限多 20 秒**——這是預留 buffer、避免任務還在跑就被重派、變成同一個任務做兩次。
>
> Dead-letter queue 是失敗重試的安全網——如果 Lambda 處理一個訊息失敗 2 次、SQS 會把這個訊息送到死信佇列、不會無限重試。
>
> 下面 Lambda triggers 可看到這個 queue 直接連到 `tdcs-dl-cleaner` Lambda——**這就是 §3.2 dual dispatch 中 SQS consumer 那個入口**。」

## 步驟 10：Cost Explorer 解釋費用組成（回應「為什麼這麼便宜」疑問）

### 10.1 操作

1. 右上角點使用者名稱 dropdown → 點 **「Billing and Cost Management」**
2. 左欄找 **「Cost Explorer」** → 點進去
3. 如果是第一次開、可能需要點「Launch Cost Explorer」
4. 預設會看到「Cost and usage reports」儀表板

### 10.2 看什麼

頂端的「Cost and usage」widget：

| 欄位 | 應顯示 |
|---|---|
| Current month | 約 **$0.21** USD |
| Forecasted month end | 約 **$0.39** USD |
| 比上月百分比 | ▲ 大數字（300%+）|

下方 6 個月柱狀圖：1-5 月幾乎是 0、6 月最高。

### 10.3 講稿（這段是重點、老師質疑費用要詳細講）

> 「老師可能覺得『$0.21 怎麼這麼少、是不是沒在用 AWS？』——**不是、便宜本身就是工程價值的證明**。
>
> **首先、$0.21 怎麼來的？**——是 6 月 1 號到 6 月 13 號累計、AWS 帳單系統實際算出來的。組成分項是：
>
> | 服務 | 金額 | 為什麼 |
> |---|---|---|
> | **Amazon S3** | $0.18（85%）| 17 GB 資料 × 13 天 prorated 儲存費 |
> | **CloudWatch** | $0.019（9%）| Lambda 497 次呼叫寫的 log |
> | **ECR** | $0.009（4%）| Lambda container image 存放 |
> | 其他 | $0.005（< 1%）| API Gateway / Athena / EC2 metering 等 |
>
> 上面這些**老師可以等等在 Cost Explorer 點 Group by Service 看到**——數字對得上。
>
> **再來、為什麼這麼便宜？**——因為我們選了 **serverless 架構**：Lambda 是按呼叫次數計費、Athena 是按 scan 大小計費、SQS 跟 API Gateway 都有 1 百萬次免費額度、我們才用了幾百次。**沒有任何一個服務是『放著不關還會跑錶』的常駐機器**。
>
> 如果改用 EC2 開一台機器 24 小時跑、或 RDS 開一個資料庫、月費直接跳到 $30-50 美金。我們是 $0.21、**便宜 200 倍以上**——這就是書面報告 §3.5 + §7 講的『對的工具解對的問題』、便宜本身就是證明。
>
> **最後、預估 $0.39 是怎麼算的？**——是 AWS 用『最近觀察到的每日花費平均』乘上 30 天。我們每日大概 $0.013-0.019、平均下來月底會到 $0.39。」

### 10.4 進階：Group by Service 看分項

如老師想看更細：

1. Cost Explorer 主畫面左欄選「Explore」→ 預設 view 是「Service」
2. 時間範圍選 Last 14 days
3. 應看到柱狀圖、Group 是 Service

## 附錄 B：學生 CLI 一鍵全驗證（如老師希望快速看一輪）

```bash
# 1. 確認身分
aws sts get-caller-identity --region us-east-1

# 2. 確認 S3 3 partition
aws s3 ls s3://112021024/raw/ --region us-east-1
aws s3 ls s3://112021024/cleaned_v2/ --region us-east-1

# 3. 確認 Lambda image tag
aws lambda get-function --function-name tdcs-dl-cleaner --region us-east-1 --query 'Code.ImageUri' --output text

# 4. 確認 Athena workgroup
aws athena get-work-group --work-group tdcs-dl-wg --region us-east-1 --query 'WorkGroup.Configuration.BytesScannedCutoffPerQuery' --output text

# 5. 確認 Glue table schema
aws glue get-table --database-name tdcs_dl --name cleaned_v2_skeleton --region us-east-1 --query 'Table.StorageDescriptor.Columns[].[Name,Type]' --output table

# 6. 當場跑 Athena 查詢
aws athena start-query-execution \
  --query-string "SELECT yyyymm, COUNT(*) AS rows, SUM(counts) AS total_traffic FROM tdcs_dl.cleaned_v2_skeleton GROUP BY yyyymm ORDER BY yyyymm" \
  --work-group tdcs-dl-wg --region us-east-1
# → 拿回 QueryExecutionId
# 等 3 秒後：
aws athena get-query-results --query-execution-id <貼上面那個 ID> --region us-east-1 --query 'ResultSet.Rows[].Data[].VarCharValue' --output json
```

每個指令應該都回到對應的數字、跟 Console 操作的結果一致。
