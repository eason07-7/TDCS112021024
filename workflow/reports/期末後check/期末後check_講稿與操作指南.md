> 「先從**本地端的 CLI 工具**講起。

> **技術棧**：我們選了 TypeScript 寫程式、ink 做終端機介面、用 esbuild 打包成單一檔。

> ink 這個套件可以讓我們在終端機跑 React 元件、做出互動式選單。比起傳統一行一行問「請輸入年份」「請輸入月份」、互動式選單對使用者**友善很多**。
>
> esbuild 把所有依賴打包成單一 `dist/index.js`、使用者不必另外裝套件、節省磁碟空間和環境麻煩。

### 投影片 7 ‧ CLI ↔ AWS 對接（1.5 分鐘）

> 「這頁講**本地工具跟 AWS 怎麼對接**——這部分**書面報告其實沒寫到**、是我們系統的核心細節。
>
> **左半 endpoint 三層設計**：
>
> CLI 怎麼知道要連 AWS 哪個 API Gateway 網址？我們做了三層 priority：環境變數 > 設定檔 > 預設 hardcoded。
>
> 為什麼三層：**教學版**預設寫死、使用者不必設定就能用；**自架版**改設定檔就指到自己 AWS 的 API GW。**同一份 CLI 兼容兩種使用情境**、不用分兩個 binary。
>
> **右半觸發 + 輪詢機制**：
>
> CLI 打 POST /clean、API Gateway 秒回一個 job_id——**不卡 30 秒 timeout**。然後 CLI 每 5 秒 GET /jobs/{id} 一次、看狀態走到 done 就結束等待。狀態流程是 accepted → processing → done。
>
> **這樣設計的理由**：HTTP 是標準協定、不必建長連線、伺服器跟 CLI 都可以掛掉重來。」

### 投影片 8 ‧ Lambda Container 技術選擇（1.5 分鐘）

> 「Lambda 有兩種部署方式：**Zip 包**或**Container image**。Zip 限 250 MB、只能用 Lambda 預設提供的 runtime；Container 限 10 GB、可以自帶 runtime。
>
> **我們選 Container、原因有 4 個**：
>
> 1. 我們清洗用 nodejs-polars 這個套件、它底層是 Rust 寫的 native 模組——**不在 Lambda 預設 runtime 裡**、用 Zip 部署裝不進去。
>
> 2. Container 讓我們用 Docker 把 polars + Node.js 包成 image、推到 AWS 的 ECR（Container 倉庫）、Lambda 拉這個 image 來跑。
>
> 3. **Image tag = git commit SHA**——每次部署 tag 都不一樣、Terraform 看得到變化會更新 Lambda。**之前用 :latest 標籤 Terraform 抓不到變化、Lambda 一直跑舊版**、是我們踩過的坑、後來改成 SHA tag 解決（書面 §3.3）。
>
> 4. 配置是 memory 2048 MB、timeout 15 分鐘、暫存 1 GB——這就是書面 §2.1 配置表那一列、各位之前看到的 demo_17 就是這個 Lambda 的 Console 截圖。」

### 投影片 9 ‧ SQS broker（1.5 分鐘）

> 「這頁講**整個架構的時間鴻溝**怎麼解。
>
> 問題：**API Gateway 最多只能等 30 秒就要回應**、可是我們清洗一個月的資料要 847 秒、也就是 14 分鐘——**差 28 倍**。
>
> 如果照原本同步等清洗跑完才回的方式、30 秒一到就會被回錯誤、就算 Lambda 還在好好做事也一樣。
>
> **我們的解法是中間加一個 SQS 排隊系統**：把「接受任務」跟「真的去做」拆開——
>
> **Producer 路徑**：API Gateway 收到請求 → 寫進 SQS → 立刻回 job_id、不到 1 秒。
>
> **Consumer 路徑**：Lambda 從 SQS 接到訊息、慢慢做 14 分鐘的清洗、不必任何人等。
>
> 同一個 Lambda 掛兩個 trigger（API GW + SQS）、**程式碼一份、兩個進入點**——這在 Lambda 圈叫做 dual dispatch。
>
> 還有 **DLQ 死信佇列**——失敗重試 1 次後送到死信、不無限重跑——是 SQS 標配的安全網。
>
> 這就是書面 §3.2 那段、剛才大家看到的 demo_18 就是 SQS 設定頁。」

---

### 投影片 10 ‧ Athena + Glue 技術選擇（1 分鐘）

> 「清洗完的資料怎麼查？我們用 **Glue 加 Athena** 這組 serverless 工具。
>
> **Glue Data Catalog** 註冊資料表 cleaned_v2_skeleton、9 個欄位 + partition 是 yyyymm。Lambda 寫完 Parquet 就跑 MSCK REPAIR、自動發現新 partition、不必手動加。
>
> **Athena** 是 serverless 的 SQL 查詢服務、開查就用、沒在用就不收錢、按掃描大小計費 5 美元 / TB。我們設了 10 MB 掃描上限當護網——萬一不小心查到大資料就拒絕、避免帳單爆炸。實測**查 1 次只掃幾十 KB、計費不到 1 美分**。
>
> **為什麼不用 RDS 或 RedShift**——那些是**常駐機器、按月計費**、RDS 每月 50 美元、RedShift 每月 180 美元。我們資料量小、查詢稀疏、用不到。Athena 規模小時就是免費、規模大時才開始付費——完美 fit 學期專題。」

### 步驟 7 ‧ Athena 確認資料正確（2 分鐘）

**操作**：

1. AWS Console 上方搜尋輸入 **Athena**、點進去
2. 進「Query editor」
3. 上方 Workgroup 下拉、選 **`tdcs-dl-wg`**（不是預設的 primary）
4. 中間 Query 編輯區、貼入下面 SQL：

```sql
SELECT day, gantry_id_o,
       COUNT(*) AS rows,
       SUM(counts) AS total_traffic
FROM tdcs_dl.cleaned_v2_skeleton
WHERE yyyymm = '202606'
GROUP BY day, gantry_id_o
ORDER BY day, gantry_id_o;
```
