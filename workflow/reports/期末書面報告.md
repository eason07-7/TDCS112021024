## §1 系統定位

本專題把期中針對**國道 1 號麻豆段**所做的「下載 → 清洗 → 視覺化 → 分析」一次性手工流程，升級成一套**參數化、可重現、未來通用的命令列工具（CLI）+ AWS 雲端後端**。

**核心價值**：讓**未來任何有 TDCS 國道車流資料需求的人**——研究者、學生、工程師——**省下「自己寫程式下載原始資料 + 自己寫程式清洗髒資料」的大量時間**。過去每個人都得各自摸索 TDCS 的下載與清洗；現在只要一行指令、跟著 wizard 五步，就能拿到乾淨、可直接分析的結果。

**AWS 在其中的角色**：把「大資料的儲存與清洗運算」這些笨重的工作交給雲端——**S3** 存放原始與清洗結果、**Lambda** 執行清洗、**SQS** 串接非同步任務、**Glue + Athena** 讓清洗後資料直接用 SQL 查詢。使用者的電腦只負責「發任務、拿輕量結果」，不必承擔大資料的儲存與運算壓力。

> **一句話**：同一套工具，麻豆段能跑，未來任意國道路段也能跑。

## §2 系統架構

五件套：CLI（本機）+ Lambda（清洗）+ S3（儲存）+ Glue Data Catalog（schema）+ Athena（查詢）。

![TDCS CLI ↔ AWS 系統架構流程圖（E6 client-side 下載 + SQS 非同步 broker、含 (1)–(7) 端到端流程）](期末/系統架構圖.png)

> 圖：完整端到端架構——本機（台灣 IP）下載 TDCS → 上傳 S3 raw/ → API Gateway → SQS 非同步 broker → Lambda 清洗（polars）→ Parquet 寫回 S3 cleaned_v2/ → Glue 註冊 + Athena 查詢；DLQ 補失敗、CloudWatch 監控、IAM LabRole 授權。

關鍵設計：

- **CLI ↔ AWS API 分離**：CLI 完全可離線運作（pull 走 E6 client-side 直接相對 TDCS）、AWS 只在 clean / Athena query 時必要
- **同步 producer + 非同步 consumer**：API GW 30 秒整合 timeout vs Lambda 最長 15 分鐘清洗時間、靠 SQS broker 切兩段，見 §3.2
- **Pascal→snake schema mapping**：TDCS 原 CSV header 是 PascalCase（`GantryID_O`），Glue + Parquet 統一 snake_case（`gantry_id_o`）、避免大小寫衝突

### 3.1 重複下載判斷 — 不浪費 S3 上傳

**問題**：

這套工具的設計目標是讓**未來有 TDCS 下載需求的使用者共用同一份雲端資料**——當使用者 A 已經下載並上傳某月某匝道的原始資料、使用者 B 之後要分析同月同匝道時，理想上**不該再從 TDCS 重抓、重新上傳一次**。以單月單路段原始資料約 22 GB 計，重複十次就是 220 GB 的頻寬與儲存浪費。

**觀察**：

`tdcs-dl pull` 本質上是 raw 資料的 fetch + persist、不是計算密集任務。S3 上若已存在同 `yyyymm` + `route` 的完整 raw set（檔數對得上 TDCS manifest），**本機根本不該再抓**、直接 reuse cloud 上的就好。

**對策**：

設計層加 **dedup gate**：
1. wizard step 4「選輸出」前、CLI 先呼叫 `HEAD s3://bucket/raw/yyyymm=YYYYMM/`、count gz 檔數
2. 若數量 ≥ TDCS manifest 預期數（744 for 31 天 × 24 hr）→ 提示 user「S3 已有完整 raw、要 reuse 還是強制重抓？」
3. user 選 reuse → CLI 跳過 pull、直接走 clean、省 ~30 分鐘 + 22 GB upload

**驗證**：

dedup gate 建立在**已驗證的能力**之上、不是憑空構想：raw 已依 `yyyymm` 分區存於 S3（§3.6 client-side 上傳鏈已實證 22 GiB / 744 檔落地），而「數同前綴下的物件數、比對 TDCS manifest 預期檔數」用的是 `ListObjectsV2` / `HEAD` 這類 S3 標準能力——判斷邏輯是確定可行的。它與 §6 v3.0 的「自帶帳號自架」「streaming pull」屬同一條設計主軸：**讓上游苦工只做一次、雲端共享，使用者不重複承擔**。

### 3.4 Schema 型別漂移 — 抓 bug → root cause → 修好

> **本報告核心主線**：這條線完整展示了「**已驗證核心邏輯**的工程方法論」、從 bug 發現到 root cause 鎖定到驗證修好的三連證據鏈。

**問題**：

端到端測試跑完、用 Athena `SELECT COUNT(*)` 拿到 14,116 行對得上 baseline（通過）。但跑 `SELECT vehicle_type, SUM(counts) GROUP BY vehicle_type` 立刻噴：

```
HIVE_BAD_DATA: Malformed Parquet file.
Field vehicle_type's type DOUBLE in parquet file ...
is incompatible with type integer defined in table schema
```

![Athena GROUP BY 噴 HIVE_BAD_DATA：Parquet 內 vehicle_type 為 DOUBLE、與表定義的 INT 不相容](screenshots/E10/demo_15_bug_schema_type_drift.png)

**觀察**：

DOUBLE vs INT 型別不對齊。第一直覺：**SQL CAST 修**——`SELECT CAST(vehicle_type AS INTEGER) ...`。試了。**也噴同錯**：

![改用 SQL CAST 仍噴同一個 HIVE_BAD_DATA — 證明 root cause 不在查詢層](screenshots/E10/demo_15b_bug_cast_also_fails.png)

關鍵觀察：**Athena CAST 救不了**。因為錯誤發生在「Parquet reader 讀檔的階段」、SQL CAST 是在「資料讀進 query engine 後」才能作用、根本到不了 CAST。問題在寫檔不在查詢。

**對策**：

回頭看 Lambda 清洗端的邏輯（以下以虛擬碼描述）：

```
（清洗端，修前）
df ← 讀入清洗後的資料列          # 沒有告訴資料框各欄的型別
將 df 寫成 Parquet 檔
```

讀入時沒給 schema、資料框依推斷規則把 JS Number 通通推成 64-bit 浮點數（因為 JS Number 規範就是 64-bit float、推斷器不知道我們意圖是整數）。浮點 → Parquet 寫成 DOUBLE → Glue table 對應欄是 INT → mismatch。

修法：**寫入前明示 schema**：

```
（清洗端，修後）
定義欄位型別：
    8 個數值欄（year / month / day / weekday / hour_0 /
                vehicle_type / counts / week_index）→ 32-bit 整數
    gantry_id_o                                     → 字串
df ← 讀入清洗後的資料列（明示套用上述型別）   # 不再讓推斷器自己猜
將 df 寫成 Parquet 檔                          # 數值欄落 INT32、不再是 DOUBLE
```

**8 個數值欄 32-bit 整數 + 1 個字串欄**、推斷器不再猜、Parquet 就會落 INT32。

**驗證**：

修 + 重 deploy + 重 clean（覆蓋舊 DOUBLE Parquet）+ 重跑 Athena GROUP BY：

| Query | 結果 | Run time | Scanned |
|---|---|---|---|
| `SELECT vehicle_type, SUM(counts) ... GROUP BY vehicle_type` | **5 列 PASS、無 HIVE_BAD_DATA** | 767 ms | 32.67 KB |

![修正後重跑 Athena GROUP BY：5 列正常回傳、無 HIVE_BAD_DATA](screenshots/E10/demo_16_athena_groupby_fixed.png)

5 列結果：vehicle_type {5: 111,926 / 31: 980,484 / 32: 338,957 / 41: 10,767 / 42: 87,699}、總 1,529,833 次車輛通過。

**三連 evidence chain**：

`demo_15`（bug）→ `demo_15b`（CAST 也失敗、證明 root cause 不在 SQL）→ `demo_16`（顯式 schema 修好、Athena PASS）

**附帶效益 — Parquet 檔案縮 12 倍**：

修補的設計動機是正確性、但實際拿了一個**意外的效率紅利**。同一份清洗結果（仍是 14,116 行、同 9 欄）、僅 dtype 從 Float64/DOUBLE 改成 Int32：

| 版本 | Parquet 大小 | 行數 | dtype |
|---|---|---|---|
| 修前（demo_11）| **903.5 KB** | 14,116 | Float64 / DOUBLE |
| 修後（live 從 S3 撈）| **76,673 B**（74.9 KB）| 14,116 | Int32 |
| **縮減比** | **~12×** | — | — |

8 個 numeric 欄 Float64 是 8 bytes / value、Int32 是 4 bytes / value、加上 Parquet 對相同值的 dictionary encoding 在 Int32 上效果更好——同一份資料、寫入時多給 polars 一個 schema 提示，下游所有的儲存與掃描成本都連帶降。**明示優於推斷** 不只是正確性問題、也是效率問題。

**工程方法論收穫**：

1. **錯誤不在你以為的層**：CAST 不行是因為錯在 reader 不在 query。看完整 stack 而非只看 error message。
2. **明示優於推斷**：型別系統的「自動推斷」在跨層邊界（JS → Parquet）會出問題、跨層處要顯式宣告。同一份修正同時拿到「對」與「省」兩個結果。
3. **修完要重 deploy + 覆蓋舊資料**：S3 上舊 DOUBLE Parquet 不會自己變、要重清洗一次才有新 INT32 Parquet。

### 3.2 SQS broker pattern — 30 秒 vs 14 分鐘的鴻溝

**問題**：

清洗業務透過 API Gateway HTTP API 觸發（`POST /clean`）。但 API Gateway 的 integration timeout 有一個**改不掉的服務硬上限：30 秒**。而我們單月清洗的實測時間是 **847 秒（14 分 7 秒）**——掃 744 個 raw 檔、聚合出 14,116 行。若 `POST /clean` 同步等清洗跑完才回應，API Gateway 會在第 30 秒直接回 504，**即使後端 Lambda 還在正常清洗**。30 秒 vs 847 秒，是 28 倍的鴻溝、不是調參數能補的。

**觀察**：

第一直覺是「把 Lambda timeout 調大」——但問題不在 Lambda（它本來就設 900 秒），在 API Gateway 那 30 秒。第二直覺是「換一個沒有 30 秒限制的觸發方式」——這才對：關鍵 insight 是**把「觸發」與「執行」拆開**。使用者要的其實不是「站在原地等 14 分鐘」，而是「丟出任務、拿到一個追蹤碼、之後查進度」。這正是經典的 **producer–consumer 非同步模式**：producer 收到請求**立刻**回一個 job_id，consumer 在背景慢慢做。

**對策**：

導入 **SQS 佇列**當 broker，同一個 Lambda 掛兩個入口（dual dispatch）。同一支處理函式依「事件從哪來」分流（以下以虛擬碼描述）：

```
函式 handler(收到的事件):
    若 事件來自 SQS 佇列 → consumer 路徑：背景執行真清洗（5–14 分鐘）
    否則（來自 API GW）  → producer 路徑：立刻回 202 + job_id（秒回）
```

- **producer 路徑**（API GW）：寫 `status=accepted` → `SendMessage` 進 SQS → 立刻回 `202 {job_id}`（< 1 秒）
- **consumer 路徑**（SQS event source）：背景跑 list→clean→Parquet→Athena REPAIR → 寫 `status=done`
- 配置上用 SQS 標配護網（非自造）：`visibility_timeout=920s`（> Lambda 900s、避免任務還在跑就被重投）、DLQ `maxReceiveCount=2`（失敗重試 1 次後進死信佇列、不無限重跑）、`batch_size=1`（一個清洗任務一個 invoke）。

**驗證**：

實跑 202603：`POST /clean` 即時回 `202 + job_id`（< 1 秒、非 504）；CLI `pollUntilDone` 每 5 秒查一次 `GET /jobs/{id}`，狀態 `accepted → processing → done`；總耗時 847 秒、14,116 行落地。Lambda Console 的 Function overview 同時掛著 **API Gateway + SQS 兩個 trigger**，即 dual dispatch 的實證。

### 3.3 git SHA tag 版控 — 不再用 :latest 鎖死

**問題**：

Lambda 以 container image 部署（image 放 ECR）。最初 `image_uri` 綁 `:latest` tag。流程是：改 handler → build → push 新 image 到 `:latest` → `terraform apply`。但 `image_uri` 字串（`…:latest`）**沒變**，Terraform 的宣告式 diff 判定「無變更」、不更新 Lambda——於是 Lambda **繼續跑上一版 image**。結果就是「我明明改了程式、重部署了，Lambda 還跑舊 code」的鬼打牆，debug 時極易誤判成「清洗邏輯有 bug」（其實是 image 沒換）。

**觀察**：

`:latest` 是一個 **mutable pointer**：同一個字串、指向的內容會變，而 Terraform 只看「字串變了沒」。第一直覺是 `terraform apply -replace=…` 強制重建——但那要求部署者**每次都記得帶這個 flag**，忘了跑純 `apply` 就又靜默回舊 image，等於把正確性託付給人的記憶。關鍵 insight：**tag 本身就該是 immutable、而且與 git commit 1:1 對應**——讓「字串」隨每次部署改變，Terraform 的宣告式模型自然就偵測得到。

**對策**：

build/push 腳本改用 git commit SHA 當 tag（working tree 有未提交變更時補時間戳保唯一），寫進 Terraform 自動載入的 `image_tag.auto.tfvars`（以下以虛擬碼描述）：

```
tag ← 當前 git commit 的短雜湊（前 12 碼，例：a381a2d7a88a）
若 working tree 有未提交變更 → 在 tag 後補上時間戳（確保唯一）
把 "lambda_image_tag = <tag>" 寫進 Terraform 自動載入的變數檔
```

於是 `image_uri` 字串隨每次部署改變 → `terraform plan` 偵測到 → **in-place 更新** Lambda code。部署者只跑 `terraform apply`、不必記任何 magic flag。

**驗證**：

兩次真實部署實證機制有效：首次部署 tag = `b71afeba8f70-dirty-…`；型別修正（§3.4）後第二次部署 tag = `a381a2d7a88a-dirty-20260604093928`，`terraform plan` 顯示 **0 added / 1 changed / 0 destroyed**（純 `image_uri` in-place、無 destroy/recreate），Lambda 約 10 秒更新完。注意 tag 前綴 `a381a2d` **正是那次型別修正的 git commit SHA**——部署在跑的 image 與原始碼版本 1:1 可追溯，這正是 `:latest` 永遠做不到的。

### 3.5 Lambda > Glue ETL trade-off — 對的工具解對的問題

**問題**：

清洗引擎選型。AWS 上兩個方向：**AWS Glue ETL**（Spark、企業級批次、課程與教科書的「大數據招牌」）vs **Lambda**（輕量、短任務）。直覺上「大數據」就該配 Spark。但問題該問的是：用一個 Spark cluster 去殺我們這個量級的資料，是合理、還是 overkill？我們的單月輸入是 **~22 GiB raw（744 個 csv.gz）**，清洗後聚合成 **14,116 行**。

**觀察**：

看清楚 transform 的形狀才知道該用哪個工具。我們的清洗是：逐檔 gunzip → 篩目標 gantry → 依（時 × 門架 × 車種）groupby 計數 → 加週次索引。**全程沒有跨檔 join、沒有 shuffle、沒有跨節點協調**。Spark 的價值正在於 join/shuffle/跨節點的分散式運算——這裡一個都用不到。關鍵 insight：這是 **Lambda 的 sweet spot**（短任務 < 15 min、簡單 transform、零叢集管理），不是 Glue 的；硬上 Spark，叢集初始化與排程的固定開銷反而可能比清洗本身還久。

**對策**：

鎖板「**純 Lambda 處理層**」：單一 TS 清洗模組（< 300 行）、`nodejs-polars`、Lambda 2048 MB（≈ 1.1 vCPU）、streaming 一次處理一個檔控制記憶體峰值。不引入 Glue / Step Functions。報告誠實寫「為什麼**不**選 Glue」——這是工程判斷、不是能力不足。

**驗證**：

實測單月清洗 **847 秒（14 分 7 秒）**、輸出 14,116 行，落在 Lambda 900 秒 timeout 內。值得一提的是「知道怎麼更快、但有意識地不做」：Lambda 的 CPU 隨記憶體線性配（調到 10240 MB ≈ 6 vCPU 會更快），但瓶頸還包含 22 GiB 的 S3 下載 I/O + 逐檔單線程處理，加記憶體報酬遞減；單月 14 分鐘已在配額內，為省幾分鐘而上平行/Step Functions 架構反而是過度工程。這正是 §0.3「工程合理性——不過度」的具體落實：對的工具解對的問題。

### 3.6 E6 client-side relay winner — 七種跳板的科學選擇

**問題**：

TDCS 公開資料（`tisvcloud.freeway.gov.tw`）**只接受台灣 IP**，而我們的 AWS backend 鎖死 us-east-1（Learner Lab 強制）。「怎麼讓 us-east-1 抓到只給台灣 IP 的資料」是整個架構最大的洞——它決定產品能不能 cloud-native。

**觀察**：

不靠拍腦袋，設計 **7 個對照實驗（E1–E7）× 4 指標（latency / cost / reliability / deploy 複雜度）**。先用 E1（Lambda 直連）+ E2（EC2 直連）建立基線：兩者打 TDCS 都是 **silent drop**——E1 timeout 15,990 ms、E2 `curl exit 28` / `http_status 000`（連 TCP 握手都沒完成）。**實證封鎖是 region-level**（us-east-1 整段 IP），換 NAT/服務類型都沒用。所以一定要繞。候選方案各有代價：E3 第三方 proxy（$4–16/月 + 信用卡 + 爬蟲條款風險）、E4 Tailscale（3 個 SPOF + $0.09/GB egress + ~55 min setup）、E5 nginx + Cloudflare tunnel（免費但 user 機 SPOF + quick tunnel URL 不穩）、E7 hybrid webhook（5 個依賴鏈）。關鍵 insight：**下載必須從台灣 IP 出去，是物理約束、不是設計選擇**。

**對策**：

選 **E6 純 client-side**：使用者本機（台灣 IP）直連 TDCS 下載 → gzip → 上傳 S3；AWS 只負責「收檔 + 清洗 + 查詢」這些真正的重活、不負責去「拉」（也拉不到）。這同時化解了「E6 把下載放本機是否違反『AWS 做重活』」的疑慮——重新框定後答案很清楚：**下載是台灣 IP 物理約束、清洗與查詢才是重活，而那兩者都在雲端**。

**驗證**：

概念驗證已通過（走完整 download → gzip → S3 鏈）；正式階段真跑麻豆段 **22 GiB / 744 檔 / 43 分鐘 / 6 狀態任務追蹤**、零跳板費用。E1/E2 的 timeout 數據就是「為什麼非繞不可」的硬證據——這是把一個架構決策建立在實測對照上、而非假設。

> **設計延伸（streaming pull、概念設計）**：目前 pull 是「整月下載到本機 temp → 全數上傳 → 刪除」，峰值本地佔用 ~22 GiB（任務結束後歸零）。下一步設計是 **streaming pull**——邊下載、邊上傳、邊刪除（HTTP → gzip → S3 串流、不落地），讓本機峰值佔用恆定在 ~1 個檔、且中斷不留殘留。這把「使用者電腦不需承擔大資料存儲」從「長期歸零」推進到「過程中也趨零」，與清洗端 Lambda 既有的「一次一檔 streaming」設計一致（同一套資源恆定哲學、客戶端與伺服端對齊）。

### 3.7 Wizard 輸出落點 — 重活在雲、結果由 user 自選

**問題**：

CLI 跑完，資料該放哪？這背後其實是更根本的問題：**使用者的電腦該承擔多少？** 一個月 raw 是 ~22 GiB、清洗要十幾分鐘的 CPU；若把「存 22 GiB」和「跑清洗」都壓在使用者本機，等於要求每個研究者的筆電都先有大硬碟 + 長時間燒 CPU——這正是大多數人卡在「前置苦工」的原因。

**觀察**：

關鍵是把「**重活的落點**」和「**結果的落點**」分開：
- **重活**（22 GiB raw 存儲、清洗 compute）→ 必須在雲端。本機扛不起、也不該扛。
- **結果**（清洗後的輕量 Parquet，單月僅 ~76 KB）→ 才讓使用者選要不要拉回本機。

於是 wizard 第 4 步「輸出位置」提供三選項：**本機 / S3 only / 兩邊**，對應不同工作流（demo 只要雲端、本機開發要落地、混合則兩邊）。

**對策**：

架構上「重活恆在雲」是已落地的事實：raw 上 S3、清洗在 Lambda、本機下載用的 temp 上傳完即刪（本機**長期零存儲**）。輸出落點則由使用者於 wizard 自選（`cli/src/wizard/state.ts` 的 `OutputDest = 'local' | 's3' | 'both'`、`Output.tsx` radio 選單），不寫死、不強迫兩邊都存。

**驗證**：

核心價值「**雲端扛存儲 + 清洗、本機保持輕量**」已端到端驗證：單月 22 GiB raw 存於 `s3://112021024/raw/`、清洗在 Lambda 跑 847 秒、本機 temp 清空、結果 `cleaned.parquet` 落於 `s3://…/cleaned_v2/yyyymm=…/`（Athena 查得到 14,116 行）。三選項中 **S3 輸出已驗證**；**「本機儲存 → `./tdcs-output/`」目前為設計（wizard 已具該選項與 UI），將清洗結果下載回本機的實作列為後續完善**——本報告誠實標示其實作邊界，不宣稱本機已落檔。

### 3.8 資料正確性的誠實驗證 — 為什麼是 14,116 行而非 14,058

**問題**：

驗收時原本想拿清洗結果對期中階段的 **14,058 行**做完全比對。但這次實際清洗出 **14,116 行**——多了 58 行，逐位元組比對必然對不上。要不要為了「數字對齊」而硬湊？

**觀察**：

先查根因、再決定。+58 行**不是 bug**：這次抓齊了該月份完整的 744 個逐時檔，而期中階段當時只有 741 檔（漏抓 3 個小時），3 小時 × 約 20 行 ≈ +58，數字完全對得上。關鍵：既然**輸入資料本來就不同**，「對 14,058 做完全比對」這個指標本身就不合理——硬湊就是自欺。

**對策**：

不降品質，改用更誠實的三層驗證取代逐位元組比對：
1. **邏輯同源**：下載端與清洗端共用同一份清洗程式碼、邏輯完全一致
2. **行數可解釋**：14,116 行，且 +58 的來源已查清（多抓 3 個逐時檔）
3. **schema 對齊**：清洗輸出的 9 個欄位型別與查詢表完全對齊（修正後皆為整數，見 §3.4）

**驗證**：

這不是「先湊過再說」，而是把「為什麼接受這個差異、差異從哪來、之後如何補強完全比對」明文記錄、並排入後續版本的實證項目。**誠實面對資料差異、解釋清楚、留下補強路徑**，比硬湊一個漂亮但站不住的數字更有價值——這本身就是工程方法論的展示。

## §6 v3.0 設計：自帶帳號、一次苦工、全民共享

本報告示範的是 **demo / 教學版**——用 AWS Learner Lab 展示工程方法論，有兩個本質侷限：(1) Lab 帳號 4 小時 session、學期結束收回；(2) 共用同一個 bucket 與憑證，無法安全地開放給多人或長期使用。v3.0 的設計主軸就是把這兩個侷限拆掉，而**核心不變**：同一份 wizard、同一份清洗邏輯、雲端依舊承擔大資料的存儲與清洗、使用者的電腦保持輕量。以下五項設計，是 v2 已驗證核心之上的自然演進。

**1. 自帶 AWS 帳號 + IaC 一鍵自架（取代共用金鑰）**

交付物是 **CLI + Terraform IaC**，不是一把要分享的鑰匙。使用者拿**自己的** AWS 帳號跑一次 `terraform apply`，就有**自己的** S3 / Lambda / Athena / SQS 後端，用**自己的**金鑰（以下以虛擬碼描述）：

```
git clone <repo> → 進入 infra/terraform 目錄
terraform apply        # 換成你自己的 AWS 帳號 → 建出你自己的後端
設定 CLI 的 endpoint 指向「你自己的」API Gateway 網址
```

誰都不必跟我們拿金鑰，彼此的額度、資料、安全邊界完全隔離——這同時解掉教學版的「共用 bucket」與「公開 endpoint 濫用」兩個問題。配置全在 Terraform（§2.1），複現只是一個指令。

**2. 重複下載判斷（dedup）**——見 §3.1。S3 已有完整 raw set 就跳過 TDCS 下載 + 上傳，省 ~30 分鐘 + 22 GiB 流量。「上游苦工只做一次、雲端共享」的關鍵節能。

**3. streaming pull**——見 §3.6。邊下載、邊上傳、邊刪除，本機峰值佔用恆定在 ~1 個檔、且中斷不留殘留；與清洗端 Lambda 的「一次一檔」streaming 一致。

**4. presigned URL 免金鑰上傳大檔**——上傳 22 GiB 不該走 API Gateway（10 MB payload 上限）。由一個端點發短效、限定路徑的 **S3 presigned PUT URL**，client 拿它直傳 S3（繞過 API GW 限制），既不持有長期金鑰、又能搬大檔。這是「clean 不需呼叫端金鑰」（§3.2）對應到「上傳」側的乾淨解。

**5. 彈性時間區間**——目前 wizard 限 ≤ 3 個月 preset（為防單一 Lambda invoke 超時而設）。v3.0 後端改「**逐月 invoke**」（每月一個 SQS 訊息 / 一個 Lambda invoke、天然 chunk、永不撞 15 分鐘上限）後，前端即可放寬成任意起訖月份。

**6. Backend portability — 同一份 CLI、兩條 fork**

上述五項是 cloud-native fork（v3.0a：自帶 AWS 帳號、保留同一套 S3/Lambda/Athena 架構）。但 v3.0 還有一條同等正當的 fork：**純本地版**（v3.0b）——把 Athena 換成 **DuckDB**（本機就能跑 SQL 查 Parquet）、把 Lambda 清洗收回本機 worker process、把 S3 收回本地 filesystem。**wizard 五步流程不變、清洗邏輯（`tdcs-clean.ts`）不變、輸出 schema 不變**——換的只是後端執行引擎。

| fork | 後端 | 適合情境 |
|---|---|---|
| v3.0a cloud-portable | 自己的 AWS 帳號（同 v2 架構） | 多人共享資料湖、團隊長期維護、可 audit |
| v3.0b local-only | DuckDB + 本地 Parquet | 個人研究、無雲端帳號、隱私敏感資料 |

兩條 fork 的設計本身就是工程價值的展示：**核心邏輯與執行平台解耦**——使用者依場景選 backend、不必為了換引擎重寫 client。這也是 v2 「Pascal→snake schema mapping + sync copy 公理」（§3.4 + §3.8 提及）所鋪的伏筆——schema 與清洗邏輯一旦穩定，誰來執行其實是 swap 的問題。

> 這六項背後是同一句話：**讓大資料的存儲與清洗壓力**留在最合適的地方執行**、一次做好給所有人共享，使用者的電腦只發任務、拿輕量結果。** v2 已證明這套設計在雲端跑得起來（§4 真數據）；v3.0 把它從「一個共用教學帳號」推進到「人人可自架的事實標準前置工具」。demo 後將開公開 GitHub repo（與教學 repo 區隔）長期維護。

### 7.2 為什麼幾乎不花錢——這是工程選擇的成本實證

成本趨近於零**不是因為沒在用 AWS、而是因為選對了服務形態**（呼應 §3.5 Lambda > Glue）：

| 設計選擇 | 對成本的影響 |
|---|---|
| **純 Lambda**（按呼叫計費、無常駐叢集） | vs Glue Spark / EMR / Redshift 的常駐叢集**省去整筆固定費**——清洗一個月只跑一次 ~14 分鐘、用多少算多少 |
| **Athena 10 MB scan cap** | Athena 按掃描量計費（$5/TB），單次查詢掃 < 30 KB → 幾乎不計 |
| **S3 標準儲存量小** | 單月 cleaned.parquet 僅 ~75 KB（型別修正 INT32 後）；raw 雖 22 GiB 但 demo 期短暫 |
| **SQS / Glue Data Catalog** | 用量在 AWS 免費額度內 |
| **無 EC2 / RDS / Redshift 常駐資源** | 零「開著就計費」的服務 |

### 7.3 節省策略

全棧採「**按用量計費 + 無常駐**」服務組合：Lambda（按呼叫）+ Athena（按掃描、加 scan cap 護網）+ S3（按儲存）+ SQS/Glue（免費額度）。要再省，只需 demo 後對 raw/ 加 S3 lifecycle 自動清理。**「對的工具解對的問題」不只跑得動、還省得下——這是 §3.5 工程取捨在帳單上的證據。**
