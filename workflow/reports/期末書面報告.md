## §1 系統定位

把 `mcp_workspace` 階段 1（2026/03 麻豆段 4 匝道 manual 流程）的「下載 → 清洗 → 視覺化 → AI 分析」四大管線，升級成**參數化、可重現、未來通用**的命令列工具（CLI）+ AWS backend。一句話：

> **同一個 CLI，麻豆段可以跑，未來任意路段也可以跑。**

階段 1 的 14,058 行清洗結果是手工跑 Python script 一次性產出、流程綁死月份 / 匝道 / 路徑。階段 2 的 v2 CLI 透過 wizard 收輸入、Lambda 真清洗、Athena 真可查、user 不需要動程式碼。

> **影片 demo**：5-8 分鐘示範 wizard → 下載 → 清洗 → S3 → Athena 全流程。連結見封面。

## §2 系統架構

五件套：CLI（本機）+ Lambda（清洗）+ S3（儲存）+ Glue Data Catalog（schema）+ Athena（查詢）。

```
┌──────────────────────────────────────────────────────────────────────┐
│                          User 端                                       │
│  $ npx tdcs-dl     ← TUI wizard（5 步：資料 / 時段 / 匝道 / 輸出 / 確認）│
│        │                                                              │
│        ▼                                                              │
│  ┌─────────────────┐  ┌──────────────────────────────────────────┐  │
│  │ E6 client-side  │  │  HTTP API Gateway                          │  │
│  │ relay：本機直接  │  │  POST /pull   POST /clean   GET /jobs/<id> │  │
│  │ 抓 TDCS 上 S3   │  │                                            │  │
│  └─────────────────┘  └──────────────────────────────────────────┘  │
│        │                       │                                      │
└────────┼───────────────────────┼──────────────────────────────────────┘
         │                       │
         ▼                       ▼ (≤30s producer)
┌─────────────────────────────────────────────────────────────┐
│                        AWS us-east-1                          │
│  S3 (112021024):                                              │
│    raw/yyyymm=YYYYMM/*.csv.gz       ← E6 client-side 上傳     │
│    cleaned_v2/yyyymm=YYYYMM/cleaned.parquet                   │
│    jobs/<job_id>.json                ← JobRecord 6-state      │
│        ▲                                                       │
│        │                                                       │
│   SQS clean-jobs queue ──────► Lambda cleaner (Node.js 20)    │
│        │  (≤920s broker)        ├─ nodejs-polars 真清洗        │
│        ▼                        ├─ schema-mapping 9 欄         │
│   SQS DLQ (maxReceiveCount=2)   └─ write Parquet to S3        │
│                                                                │
│  Glue Data Catalog: database tdcs_dl / table cleaned_v2_skeleton │
│  Athena workgroup: tdcs-dl-wg (10 MB scan cap)                │
└─────────────────────────────────────────────────────────────┘
```

關鍵設計：

- **CLI ↔ AWS API 分離**：CLI 完全可離線運作（pull 走 E6 client-side 直接相對 TDCS）、AWS 只在 clean / Athena query 時必要
- **同步 producer + 非同步 consumer**：API GW 30 秒整合 timeout vs Lambda 最長 15 分鐘清洗時間、靠 SQS broker 切兩段，見 §3.2
- **Pascal→snake schema mapping**：TDCS 原 CSV header 是 PascalCase（`GantryID_O`），Glue + Parquet 統一 snake_case（`gantry_id_o`）、避免大小寫衝突

### 3.1 重複下載判斷 — 不浪費 S3 上傳

**問題**：

教學 demo bucket（`s3://112021024`）給全班共用，若 user A 下載 2026/05 01F3019 已上傳、user B 再選同月同匝道，現有設計會**重新從 TDCS 抓一次、再上 S3 一次**。22 GB / 月 / route 的 raw、跑十次就是 220 GB 浪費上傳頻寬 + Learner Lab S3 PUT 配額。

**觀察**：

`tdcs-dl pull` 本質上是 raw 資料的 fetch + persist、不是計算密集任務。S3 上若已存在同 `yyyymm` + `route` 的完整 raw set（檔數對得上 TDCS manifest），**本機根本不該再抓**、直接 reuse cloud 上的就好。

**對策**：

設計層加 **dedup gate**：
1. wizard step 4「選輸出」前、CLI 先呼叫 `HEAD s3://bucket/raw/yyyymm=YYYYMM/`、count gz 檔數
2. 若數量 ≥ TDCS manifest 預期數（744 for 31 天 × 24 hr）→ 提示 user「S3 已有完整 raw、要 reuse 還是強制重抓？」
3. user 選 reuse → CLI 跳過 pull、直接走 clean、省 ~30 分鐘 + 22 GB upload

**驗證**：

本期 PLAN_E10 尚未實作（demo 後 v3.0 純本地版設計時納入、見 §6）、但作為**概念設計重點**寫入報告、是工程方法論的展示——**不是寫完才有價值、是設計時想到才避免後悔**。

> 對應 commit / 事件：尚未 implement、本報告作 future work 點名。

### 3.4 F-H5 schema type drift — 抓 bug → root cause → 修好

> **本報告核心 narrative**：這條線完整展示了「**已驗證核心邏輯**的工程方法論」、從 bug 發現到 root cause 鎖定到驗證修好的三連 evidence chain。

**問題**：

PLAN_E9 端到端 smoke 跑完、Athena `SELECT COUNT(*) FROM cleaned_v2_skeleton` 拿到 14116 行對得上 baseline ✅。但跑 `SELECT vehicle_type, SUM(counts) GROUP BY vehicle_type` 立刻噴：

```
HIVE_BAD_DATA: Malformed Parquet file.
Field vehicle_type's type DOUBLE in parquet file ...
is incompatible with type integer defined in table schema
```

[配圖：`workflow/reports/screenshots/E10/demo_15_bug_schema_type_drift.png`]

**觀察**：

DOUBLE vs INT 型別不對齊。第一直覺：**SQL CAST 修**——`SELECT CAST(vehicle_type AS INTEGER) ...`。試了。**也噴同錯**：

[配圖：`workflow/reports/screenshots/E10/demo_15b_bug_cast_also_fails.png`]

關鍵觀察：**Athena CAST 救不了**。因為錯誤發生在「Parquet reader 讀檔的階段」、SQL CAST 是在「資料讀進 query engine 後」才能作用、根本到不了 CAST。問題在寫檔不在查詢。

**對策**：

回頭看 Lambda 清洗端的程式：

```typescript
// infra/lambda/src/index.ts（修前）
const df = pl.readRecords(parquetRows);
df.writeParquet(tmpParquet);
```

`pl.readRecords` 沒給 schema、polars 依推斷規則把 JS Number 通通推成 Float64（因為 JS Number 規範就是 64-bit float、polars 不知道我們意圖是整數）。Float64 → Parquet 寫成 DOUBLE → Glue table 對應欄是 INT → mismatch。

修法（PLAN_E10 對應 opus_worker_2 commit `a381a2d`）：**寫入前明示 schema**：

```typescript
const PARQUET_SCHEMA = {
  year: pl.Int32, month: pl.Int32, day: pl.Int32, weekday: pl.Int32,
  hour_0: pl.Int32, gantry_id_o: pl.Utf8,
  vehicle_type: pl.Int32, counts: pl.Int32, week_index: pl.Int32,
};
const df = pl.readRecords(parquetRows, { schema: PARQUET_SCHEMA });
```

**8 個 numeric 欄位 Int32 + 1 個字串欄 Utf8**、polars 不再猜、Parquet 就會落 INT32。

**驗證**：

修 + 重 deploy + 重 clean（覆蓋舊 DOUBLE Parquet）+ 重跑 Athena GROUP BY：

| Query | 結果 | Run time | Scanned |
|---|---|---|---|
| `SELECT vehicle_type, SUM(counts) ... GROUP BY vehicle_type` | **5 列 PASS、無 HIVE_BAD_DATA** | 767 ms | 32.67 KB |

[配圖：`workflow/reports/screenshots/E10/demo_16_athena_groupby_fixed.png`]

5 列結果：vehicle_type {5: 111,926 / 31: 980,484 / 32: 338,957 / 41: 10,767 / 42: 87,699}、總 1,529,833 次車輛通過。

**三連 evidence chain**：

`demo_15`（bug）→ `demo_15b`（CAST 也失敗、證明 root cause 不在 SQL）→ `demo_16`（顯式 schema 修好、Athena PASS）

**工程方法論收穫**：

1. **錯誤不在你以為的層**：CAST 不行是因為錯在 reader 不在 query。看完整 stack 而非只看 error message。
2. **明示優於推斷**：型別系統的「自動推斷」在跨層邊界（JS → Parquet）會出問題、跨層處要顯式宣告。
3. **修完要重 deploy + 覆蓋舊資料**：S3 上舊 DOUBLE Parquet 不會自己變、要重清洗一次才有新 INT32 Parquet。

> 對應 commit / 事件：`commit a381a2d`（修）+ `commit 1a45d37`（重 deploy + 重 clean）/ 實驗紀錄 9 事件 #13+#14

### 3.2 SQS broker pattern — 30 秒 vs 8 分鐘的鴻溝

_[opus_worker M2.2 撰寫]_

問題 hook：API Gateway HTTP API 整合 timeout 硬上限 30 秒、Lambda 清洗 22 GB raw 要 8-14 分鐘、直接 HTTP 同步呼叫**必定 timeout**。

撰寫要點：
- 問題：30s vs 14min 的硬性 gap
- 觀察：把「觸發」與「執行」分開、producer 收 request 立刻回 jobId、consumer 在背景慢慢做
- 對策：SQS clean-jobs queue + DLQ（maxReceiveCount=2、visibility_timeout=920s > Lambda 900s）、Lambda 同時掛 API GW（producer 路徑）+ SQS event source（consumer 路徑）
- 驗證：實際跑 14116 rows / 847 秒、API GW 立刻回 `{ jobId, status: accepted }`、CLI 輪詢 `GET /jobs/{id}` 5 秒一次直到 done

引述：`infra/terraform/sqs.tf` + `infra/lambda/src/index.ts` dual dispatch + 實驗紀錄 9 事件 #6 (M4.5)

### 3.3 F-H4 git SHA tag versioning — 不再用 :latest 鎖死

_[opus_worker M2.2 撰寫]_

問題 hook：Lambda container image 一開始用 `:latest` tag、改了程式 push 上 ECR 後 terraform plan 認為 image_uri **沒變**、Lambda 不重新拉 image、修補沒生效。

撰寫要點：
- 問題：`:latest` 是 mutable pointer、Terraform 看不到內容變化
- 觀察：image tag 應該 **immutable**、與 git commit 1:1 對應
- 對策：`build_and_push.sh` 用 `git rev-parse --short=12 HEAD` 當 tag、寫進 `auto.tfvars`、Terraform 看到 image_uri 變動就 plan in-place update
- 驗證：F-H5 修補時 image tag = `a381a2d7a88a-dirty-20260604093928`、terraform plan `0/1/0` 純 in-place、Lambda 10 秒 update 完

引述：`infra/lambda/build_and_push.sh` + 實驗紀錄 9 事件 #3 (F-H4 first deploy) + 事件 #14 (F-H5 second deploy)

### 3.5 Lambda > Glue ETL trade-off — 對的工具解對的問題

_[opus_worker M2.2 撰寫]_

問題 hook：要選 ETL 引擎、AWS 大方向兩個候選：Lambda（簡單）vs Glue ETL（Spark、企業級）。教科書都教 Glue、但教科書沒考慮我們的資料量。

撰寫要點：
- 問題：用 Spark 殺一個月 9 GiB data 是 overkill 還是合理？
- 觀察：Lambda sweet spot = 短任務（< 15 min）+ 簡單 transform（無 join / 無 shuffle）+ 低管理成本
- 對策：22 GB raw / 月 → groupby（hour × gantry × vehicle）→ output 14116 rows、5-14 min、完全在 Lambda 配額內。Glue 引擎初始化 + Spark cluster 開銷反而比清洗本身久
- 驗證：實測 Lambda nodejs-polars 14 min 7 秒清洗 22 GB → 14116 rows、無工程複雜度（單一 ts 檔 < 300 行）
- §0.3 工程合理性紀律的具體應用

引述：MASTER_PLAN §0 鎖板表 M4.2 + 實驗紀錄 9 事件 #11 (M6 端到端 smoke)

### 3.6 E6 client-side relay winner — 三跳板實驗的科學選擇

_[opus_worker M2.2 撰寫]_

問題 hook：TDCS 原始檔在交通部 ftp/http server、AWS Learner Lab 無法直接 outbound 抓（IAM 限制）、要 relay 一下。三個方案：E3 EC2 中繼、E4 Cloudflare Worker、E6 client-side（user 端走）。

撰寫要點：
- 問題：哪個方案最簡、最便宜、最不破 Learner Lab 配額
- 觀察：三跳板都試過、跑實驗紀錄事件對比
- 對策：E6 client-side 勝出——user 本機就有 internet、走 TDCS HTTP → 直接 PUT 進 S3 multipart upload、AWS 端只負責收檔不負責去拉
- 驗證：PLAN_E6 baseline 已驗、PLAN_E8 production 跑 22 GB raw / 43 min / 6-state JobRecord、無 outbound 帳號費

引述：`workflow/reports/brief_relay_experiments_2026-06-02.md` + 實驗紀錄 6/7/8 事件對比

### 3.8 §0.3 四步妥協機制 — 永久債避免

_[opus_worker M2.2 撰寫]_

問題 hook：工程現實不可能永遠 100% perfect、總會遇到「時間壓力 vs 品質」要妥協的時候。

撰寫要點：
- 問題：PLAN_E9 M6 baseline 三證原訂用 byte-level md5 對齊 14,058 行 baseline、實際拿到 14,116 行、md5 必然不對
- 觀察：源頭差 3 個 hourly 檔案（v2 raw 744 檔 vs v1 baseline 741 檔）→ 不是 bug、是 data correction、md5 對齊本身不合理
- 對策：pivot 改 **structural 三證**（sync copy axiom + row count + Athena schema）、byte-md5 延到 PLAN_E11 補實證 gate
- 驗證：workflow CLAUDE.md §0.3 提供「4 步妥協補救流程」：(1) 妥協明文寫紀錄 (2) PLAN 追加未來修補項 (3) MASTER_PLAN 風險登記 (4) 下個 PLAN 啟動時 Lead review pending
- 這次 pivot 完整跑了 4 步、不是 throwaway hack、是工程紀律的具體展示

引述：實驗紀錄 9 事件 #12 + workflow/CLAUDE.md §0.3 + MASTER_PLAN §7 風險表

## §5 與 mcp Stage 1 manual 流程對比

> 階段 1（2026 年 4-5 月）vs 階段 2（2026 年 5-6 月）量化對比、體現工程價值。

| 維度 | Stage 1 manual | Stage 2 v2 CLI | 提升 |
|---|---|---|---|
| 重現性 | Python script + 硬編 path、跨機器重跑要改 5+ 處 | `npx tdcs-dl` 一鍵、wizard 五步 | 無腦可重現 |
| 通用性 | 綁死 2026/03 + 麻豆 4 匝道 | wizard 收年月 + 任意匝道 list（339 個 gantry） | 從 1 case → ∞ cases |
| user 介入 | 用 Python、要懂 venv / argparse / pandas | npm CLI、不需要 Python 知識 | 對 non-coder 友善 |
| 清洗時間 | 手 trigger Python、~10 min 但要 user 監看 | wizard 觸發 / Lambda 自跑、user 不必盯 | 釋放 user 時間 |
| 證據鏈 | Python output csv、無歷史 | Athena query 可重複、S3 永久存 Parquet、Glue catalog | 可審計 |
| 工程文檔 | scratch notes | 7 PLAN + 14 實驗紀錄 + MASTER_PLAN + workflow CLAUDE.md | 可傳承 |

> 工程價值的本質不在「跑得快」（清洗時間差不多）、在「**可重現 + 可通用 + 可審計 + 可傳承**」。

## §6 Post-demo v3.0 規劃預告

> **本報告所述系統是 demo / 教學版**、用 AWS Learner Lab 展示工程方法論。但教學版有兩個本質侷限：
>
> 1. Learner Lab 帳號 4 小時 session、學期結束會被收回
> 2. 全班共用 bucket、scale 無法承擔長期維護

故規劃 **v3.0 純本地 npm 版**（PLAN_E11+）：

| 變化 | demo / 教學版 (v2) | v3.0 純本地 |
|---|---|---|
| backend | AWS Lambda + S3 + Glue + Athena | 完全本機（DuckDB + 本地 Parquet） |
| 安裝 | 需 AWS credentials | `npm install -g tdcs-dl` 即可 |
| 查詢 | Athena SQL | DuckDB SQL（同語法、無雲端依賴） |
| 適用 | 課程展示、學期內 demo | **長期維護**、社群可用 |

v3.0 不是 v2 的取代、是不同情境的選擇——v2 證明了「**這套設計搬上雲端跑得起來**」、v3.0 證明「**這套設計也可以完全本機**」、兩者背後**同一份 wizard + 同一份清洗邏輯**。

> demo 後新開公開 GitHub repo（與本教學 repo 區隔）、v3.0 從那邊長期維護。
