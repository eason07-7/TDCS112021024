## [2026-06-02 22:50] 事件 #1 [Lead] Phase 0 — workflow tooling fix：`close_plan.py` UnicodeDecodeError 修復 + INDEX 雙重 regen 收斂

### 背景

PLAN_E6 封存時 `close_plan.py` 噴出 `UnicodeDecodeError: 'utf-8' codec can't decode byte 0xb9` traceback。雖然檔案搬移 + INDEX regen 都正確完成（檔案 ops 走 shutil、不靠 stdout），但 traceback 印在前讓使用者誤以為失敗、且 `archive.py` 在每次 `archive_plan` / `archive_log` 結尾各觸發一次 INDEX regen、close_plan 串連兩次 = 雙重 regen 浪費。User 拍板「先修好 bug 再把封存邏輯弄好」。

### 目標

1. 根治 UnicodeDecodeError：close_plan parent 抓 subprocess.run 子進程（archive.py）stdout 時 utf-8 decode 失敗
2. 收斂 INDEX regen：兩次 archive 都跳過、close_plan 結尾統一 regen 一次
3. Bash/PowerShell terminal mojibake → 至少做到 stdout pipe 內容是合法 utf-8（終端 codepage 是 user 端事、script 不該強塞 chcp）

### 處理步驟

1. **Root cause 定位**：`sys.stdout.reconfigure(encoding='utf-8')` 對 pipe stdout 不可靠 — child Python 報告 encoding=utf-8、但實際 write bytes 仍 cp950（用 hex dump 驗證後 case-by-case 才確認 `e4b8ade69687` = 「中文」UTF-8 確實被正確輸出、之前看到的 mojibake 是 my Bash cp950 terminal 顯示問題、不是 stdout pipe 問題）
2. **`_force_utf8_io()` helper**：在 `close_plan.py` 與 `archive.py` main 入口呼叫；用 `io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8', errors='replace', line_buffering=True)` 重綁 stdout/stderr、TTY+pipe 皆有效（reconfigure 在 pipe 不可靠 → 換 wrap）
3. **`subprocess.run` 強化**：parent 抓子進程 stdout 時加 `env={..., PYTHONIOENCODING='utf-8'}` 與 `errors='replace'` 雙保險
4. **`archive.py` 加 `--no-regen` flag**：CLI 接口向後相容（default = False = 仍 regen、手動模式不受影響）；`archive_plan(path, regen=True)` / `archive_log(path, regen=True)` 多一個 keyword 參數
5. **`close_plan.py` 用 `--no-regen`**：兩次 `archive_one` 都傳 `--no-regen`；新增 `regen_index(workflow_dir, dry_run)` 函式、結尾統一 regen 一次（import index.py 內 `regenerate()`）
6. **docstring 同步**：4 步流程 → 5 步流程（加「regen INDEX 一次」末步）
7. **驗證**：用 `tempfile.mkdtemp` 建假 workspace（含中文檔名）+ 設 `PYTHONIOENCODING=cp950` 模擬最糟環境、dry-run 與 real run 兩跑：
