## [2026-06-04 19:30] 事件 #13 [opus_worker_2] Phase 3 — F-H5 修補：Lambda Parquet INT schema（PLAN_E10 demo 抓、PLAN_E9 後續補救）

> 寫在已封存的 logs/實驗紀錄9.md（Lead 指示）：F-H5 是 PLAN_E9 交付的 Parquet 寫入 bug、PLAN_E10 demo 階段才暴露、屬 PLAN_E9 後續補救、記在 E9 narrative 內。

### 背景

PLAN_E9 M6 demo、User 跑 Athena `SELECT vehicle_type, COUNT(*) ... GROUP BY` 撞 `HIVE_BAD_DATA: Field vehicle_type's type DOUBLE in parquet file is incompatible with type integer defined in table schema`。證據：`workflow/reports/screenshots/E10/demo_15_bug_schema_type_drift.png` + `demo_15b_bug_cast_also_fails.png`（後者證 SQL `CAST AS INTEGER` 救不了——錯誤在 Parquet **read** 階段、非 SQL 階段）。User 拍板嚴格走 root cause、不採 B（改 glue schema 將就）。

### 目標

修 `runCleanFlow` 的 `pl.readRecords(parquetRows)`：強制 8 個 numeric column 為 Int32（對齊 glue.tf `int`）、gantry_id_o 維持 string、讓 Parquet 寫出的 physical type 與 Glue table schema 一致。

### 處理步驟

1. **Root cause 確認**：`pl.readRecords(parquetRows)` 無 schema → JS Number 一律推 **Float64** → Parquet 落 **DOUBLE** → Athena 讀時對 Glue `vehicle_type int` type mismatch。CAST 在 SQL 階段、但錯誤在 Parquet reader 解析 physical type 時就拋、CAST 到不了。
2. **查 nodejs-polars 0.10 API**（決定最乾淨修法）：
   - `readRecords(records, { schema: Record<string, DataType> })` overload 存在（`bin/io.d.ts`）
   - `pl.Int32` / `pl.Utf8` DataType 存在（`bin/index.d.ts`）
   - `df.withColumns([pl.col(c).cast(pl.Int32)])` 也存在
3. 加 module const `PARQUET_SCHEMA`（9 欄、8×Int32 + gantry_id_o Utf8、註解標明 source = glue.tf）
4. 改 `readRecords(parquetRows, { schema: PARQUET_SCHEMA })`
5. test mock 加 `Int32`/`Utf8` marker + 1 個 F-H5 test（驗 readRecords 收到 schema、8 numeric=Int32、gantry_id_o=Utf8）

**修補方向 trade-off（3 選 b）**：

| 方案 | 評估 | 結論 |
|---|---|---|
| (a) `withColumns([col.cast(Int32)])` | 先讀成 Float64 再 cast 回 Int32 = 補救推錯的型、多一 pass、讀起來像「修錯誤」非「宣告意圖」 | 否 |
| **(b) `readRecords(.., {schema})`（採用）** | **讀入即宣告 9 欄正確 dtype、單 pass、無推斷依賴、schema 物件 1:1 對映 glue.tf 一眼可審**；§0.3 邏輯正確性最直接 | **採用** |
| (c) toParquetRows 內 wrap Int32 | 紅線（不改 schema-mapping）＋ JS 無 Int32 type 可 wrap | 否 |

對齊「sync copy 設計、不破 PLAN_E6 baseline」：**type 修正只在 polars 寫入層**、未動 schema-mapping.ts（PascalCase→snake_case 對映不變）、未動 tdcs-clean.ts（清洗邏輯不變、row 值不變、只改 Parquet physical type）→ baseline row 內容不變、僅 column physical type DOUBLE→INT32（這正是要修的）。
