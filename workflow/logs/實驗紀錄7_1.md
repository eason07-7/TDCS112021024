#### B. AWS 正確角色 reframe（回應 User「AWS 該怎麼正確搭配」）

**核心洞察**：別把 AWS 想成「做重活的地方」(compute)——你的資料不重（8min/月、Lambda 大材小用），硬撐 compute 故事才會卡。**AWS 在本命題的不可替代角色 = 「共享、持久、可 SQL 查詢的 cleaned 資料湖 + 查詢層」**，這是 laptop 當不了的：

| 層 | 正確角色 | 為何 laptop 取代不了 |
|---|---|---|
| **S3** | 湖體：一份共享 / 持久 / partition 好的全台 cleaned TDCS | 不想每個研究者各存 17-70 GiB |
| **Athena** | 查詢層：免下載 SQL 查 60M 行（ai_workspace 已證 1500x scan reduction）| laptop 查不動、也不該人手一份 |
| **Glue Catalog** | 讓湖可被 Athena 查、自動 partition discovery | — |
| **Lambda** | **批次 ingestion worker**（維護湖、非「即時清洗你剛上傳那份」）| 清一次給全社群、非每台 laptop 各清一次 |

**關鍵推論：你的差異化命題本身就需要 AWS 當共享層。** 「替全社群解決一次」邏輯上 = 清洗+host+查詢必須集中 = 必須在雲上。越想做 §1.3「事實標準前置層」、AWS 越是 load-bearing。**這是工程驅動、不是為分數——分數是副產品。**

**這個 reframe 一次解掉三個目前最弱的點（全是邏輯正確性問題、非分數問題）**：
- **F-H1（timeout）自然消失**：Lambda 框成「批次、非同步、逐月灌湖的 ingestion worker」、8min/月完全 OK（背景 backfill、非 user 等待）；多月 = 迴圈逐月 job、不塞單一 invoke、不必違反 lock-board 加 Step Functions。
- **冗餘消失**：現設計「user 上傳自己 raw → Lambda 即時清洗 → job_<id>/ → user 等」是冗餘（raw 本來就在本機、可本機清）＋ 像私人 scratch、不是共享湖。改成「ingestion worker 維護共享 canonical 湖」後，Lambda compute 變成不可替代（替所有人清一次）。

#### C. 課程後永續：先別問 AWS-vs-NAS、先問「host 資料 vs 發布工具」

**修正前提**：(1) 現在的 AWS 是 Learner Lab、課程後**收回**、「續用 AWS」= 開自費帳號（刷自己的卡）。(2) **TDCS 是公開資料、你的工具本身能重新生成它** → 永久 host 那份資料邊際價值低；**折舊最慢的成果是「工具 + 清洗方法 + IaC」、不是一台要養一輩子的伺服器**。

**真正的分岔**：

| 模式 | host 什麼 | 後面的人怎麼用 | 你的長期負擔 |
|---|---|---|---|
| **(a) 發布工具**（建議）| 什麼都不 host | 跑你的 npm CLI + IaC、用自己機器/AWS | **$0、零維運、你失聯也不死** |
| **(b) 自架共享湖** | always-on 可查資料 | 直接查你的湖、免下載 | 一直養（錢或硬體 + 維運）|

**建議先選 (a)**：$0 永遠、不因你畢業/卡過期而消失、正是 §1.4 release 版已規劃、endpoint 抽象(D6)就是為此設計、資料公開可被工具重建。**把資料湖當課程 demo 展示品；交付社群的真成果是工具+IaC。**

**若「就是想要」一個自己長期跑的湖、才輪到 AWS vs NAS**：

| 維度 | 自費 AWS（S3+Athena+Lambda） | NAS（Synology/QNAP）|
|---|---|---|
| 公網可達 | ✅ 內建 | ⚠️ 家用 NAT/動態 IP、要 DDNS/Cloudflare Tunnel/QuickConnect（又一次跳板、SPOF 在家網路+電）|
| 查詢層（免下載 SQL）| ✅ Athena serverless（NAS 最難複製的殺手鐧）| ⚠️ 無現成；**DuckDB** 直查 Parquet 可補 ~80%（要自己包一層）|
| 耐久性 | ✅ S3 11 個 9 | ⚠️ RAID 只防單碟；火災/失竊/勒索=全毀除非異地備份 |
| 可用性 | ✅ 99.9%+ | ⚠️ 停電/斷網就掛 |
| 成本模型 | **OPEX** ~50GiB 低流量 **$2-5/月、永遠**（3年~$70-180）| **CAPEX** 一次 **$300-500** + 電費~$2-4/月（3-5年回本）|
| 維運 | 幾乎零（managed）| 要顧 OS/磁碟/隧道 |
| 失控風險 | ⚠️ 公開 endpoint（F-H3）被掃/濫用→**帳單噴**、自費帳號**必設 billing alarm + body limit + API key** | ✅ 無帳單爆炸、最壞頻寬被吃 |

**關鍵技術點**：NAS 路線**必配 DuckDB**（直查 Parquet、免架伺服器、還原大半 Athena「免下載 SQL」體驗）；否則 NAS 退化成檔案下載站、失去「免重複苦工」命題。

**最聰明組合（給預算敏感學生）**：**NAS 當私有大倉 + DuckDB 做自己後續研究；對外發布 npm + IaC 讓社群各自自架。** 不扛公開 AWS 帳單、不扛公開 host 脆弱性、NAS 又是通用家用伺服器（CAPEX 不浪費）、社群照樣用你的工具。想偶爾對外 → Cloudflare Tunnel 臨時掛 NAS DuckDB API、不必 always-on。

**§0.3 對齊**：對公開可重生的資料、永久付費 host = 過度工程；成果是方法+工具就交付方法+工具；Athena「免下載 SQL」在課程 demo + 短期對外時真的值、長期則 NAS+DuckDB 更省更自主且無帳單失控風險。
