## 1. 本階段核心目標

> **把四大管線包裝成 MCP 伺服器，讓 Claude Desktop / Claude Code 能用自然語言調用，達到「對話式資料科學」。**

**行為鐵則（guardrail）**：**AI 每次被調用 MCP 時，第一步一定是呼叫 `get_status()` 讀自檢檔 `manifest.json`**，確認目前可用的時間區間與 Gantry 清單後，才決定下一步。避免 AI 在缺資料時亂編答案、或重複觸發已完成的下載。

- 使用者說「幫我看 2026/03 的麻豆車流」→ AI 先 `get_status()` 確認 2026/03 資料齊全、4 Gantry 都在 → 才呼叫分析工具
- 使用者說「3/26 怎麼看起來怪怪的」→ AI 先 `get_status()` 看 3/26 是否有資料 → 再呼叫 `analyze_anomaly`
- 使用者說「幫我看 2025/07 的車流」→ AI 讀 `get_status()` 發現沒有 2025/07 資料 → 主動告訴使用者「目前只有 2026/03，是否要觸發下載 2025/07？」走人在迴路確認

### Phase 2 ‧ 自檢檔 `manifest.json`（AI 行為第一步）

**目標**：建一份隨 AI 流程更新的 JSON，**AI 每次呼叫 MCP 工具前必讀這份**，確認目前有哪些時間區間、哪些 Gantry (N/S) 可用。這是對話式 AI 不亂跑的 guardrail。

- 2-1. 設計 `mcp_workspace/manifest.json` schema（**新增 `available_periods` 與 `available_gantries` 兩個專給 AI 快速讀的欄位**）：
  ```json
  {
    "last_updated": "2026-04-16T10:30:00+08:00",
    "available_periods": [
      {"year": 2026, "month": 3, "days_covered": "01-31", "days_missing": ["2026-03-26T15", "2026-03-26T16", "2026-03-26T17"]}
    ],
    "available_gantries": [
      {"id": "01F2930N", "direction": "N", "label": "下營系統→新營"},
      {"id": "01F2930S", "direction": "S", "label": "新營→下營系統"},
      {"id": "01F3019N", "direction": "N", "label": "麻豆→下營系統"},
      {"id": "01F3019S", "direction": "S", "label": "下營系統→麻豆"}
    ],
    "artifacts": {
      "raw_csv":  {"count": 741, "path": "step0_s3_download/raw_202603/"},
      "cleaned": {"monthly": "...", "weekly": [], "daily": []},
      "charts":  [{"id": "chart1_2930N", "path": "...", "type": "24h_daily_overlay", "gantry": "01F2930N", "period": "2026-03"}],
      "reports": [{"id": "analysis_202603_v1", "path": "...", "generated_at": "..."}]
    },
    "runs": [{"run_id": "...", "pipeline": "step3_ai_analysis", "status": "completed", "at": "..."}]
  }
  ```
- 2-2. 寫 `manifest_update.py` 工具：每條管線跑完自動 append；同時更新 `available_periods` 與 `available_gantries`
- 2-3. 既有四步腳本加 hook：尾端呼叫 `manifest_update.py`（不動原邏輯、只加一行）
- 2-4. **MCP `get_status()` 只回 `available_periods` + `available_gantries` + 簡要統計**，不回全部 artifacts（資訊量過大會讓 AI context 爆掉）。要細節時 AI 再呼叫 `list_charts` / `list_reports`。

**驗收**：
1. 手動跑完 Step 0-3 後，`manifest.json` 會有 4 個完成紀錄 + 9 張圖 + 1 份報告登錄
2. `get_status()` 回傳結果 < 2 KB，AI 讀完能立刻判斷「使用者要的時間/Gantry 是否可用」

### Phase 3 ‧ MCP Tools 規格設計（粗粒度優先）

**設計哲學**：**粗粒度（工作流程驅動）**，讓 AI 一次調用能完成「發現缺資料→下載→清洗→畫圖」。細粒度工具僅當粗粒度不足以回應時補上。

**規劃的 Tools**（初版清單，可增減）：

1. `get_status()` — **每次對話第一步必呼**，讀 `manifest.json` 的 `available_periods` + `available_gantries` + 簡要統計；回應 < 2 KB
2. `run_pipeline(step: str, params: dict)` — 觸發 Step 0-3 其中一步；長時任務回 `{status: "running", job_id: "..."}` 讓 AI 輪詢
3. `check_job(job_id: str)` — 查長時任務狀態（`pending / running / completed / failed`）
4. `list_charts(filter: dict)` — 依匝道 / 日期範圍 / 圖表類型過濾
5. `get_chart(chart_id: str)` — 回傳 PNG 的 base64 或 file URI（依 MCP content type 規範）
6. `get_report(report_id: str)` — 回傳 Markdown 內容
7. `analyze_anomaly(date: str)` — 封裝 z-score + Gemini 速查「某一天是否異常」
8. `gen_daily_chart(date: str, gantry: str)` — 動態產單日圖（既有腳本沒有，需新寫）

**回傳格式**：
- 圖表：MCP `ImageContent`（base64 PNG）+ 檔案路徑 metadata
- 文字報告：MCP `TextContent`（Markdown）
- 數據：MCP `TextContent`（JSON 結構）

**非同步 / 人在迴路設計**：
- 長時任務（Step 0 下載、Step 1 清洗）→ 回 `job_id`，AI 每 30 秒呼 `check_job`，或主動回「我先幫你排了下載任務，預計 10 分鐘完成，你要等還是要先看現有的？」讓使用者選
- 使用者可設「Dry-run 模式」：大動作要確認才執行（防 AI 亂跑雲端費用）

**檔案**：`mcp_workspace/mcp_server/tools_spec.md` — 每個 tool 的輸入/輸出 JSON schema。

**驗收**：規格文件寫完、可以拿給 AI reviewer 挑毛病。

### Phase 4 ‧ MCP Server 實作

**技術選型**：Python + 官方 `mcp` SDK（modelcontextprotocol/python-sdk）

- 4-1. `mcp_workspace/mcp_server/server.py` — server entrypoint、註冊 tools
- 4-2. `mcp_workspace/mcp_server/tools/` — 每個 tool 一個檔案，避免單檔肥大
- 4-3. `mcp_workspace/mcp_server/jobs.py` — 長時任務佇列（簡單 sqlite 或 dict-in-memory 皆可，初版選 in-memory）
- 4-4. Tool 實作都是**薄層 wrapper**，呼叫既有的 step0-3 腳本（不重寫邏輯）
- 4-5. `mcp_workspace/mcp_server/README.md` — 安裝、啟動、Claude Desktop 設定說明

**驗收**：`python -m mcp_server` 能啟動，`mcp inspector` 工具能列出所有 tools。
